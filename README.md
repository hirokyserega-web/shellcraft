# ShellCraft

ShellCraft — распределённая система автоматического крафта для **ComputerCraft: Tweaked** (Minecraft 1.19+/1.20+, Forge и Fabric).

Core-компьютер хранит рецепты, сканирует хранилище, строит дерево зависимостей и раздаёт задачи черепахам-воркерам параллельно. UI с вкладками рисуется на мониторе, предметы показываются по-русски, всё автообновляется с GitHub.

## Возможности

- **Автоустановка** одной командой `wget run ...`.
- **Автообновление** с GitHub без потери локальных данных (рецепты, конфиг, логи).
- **Автоопределение периферии** — сундуки, мониторы, модемы, машины подхватываются сами.
- **Core/Worker роли** + rednet-протокол (`lib/net.lua`).
- **Хранилище** предметов: несколько сундуков/бочек, pushItems/pullItems.
- **Обучение рецептов**: разложил предметы как в верстаке → система сохранила рецепт.
- **Крафт по сохранённым рецептам** через черепах-воркеров.
- **Несколько воркеров параллельно** с очередью задач, балансировкой и повтором при падении.
- **Рекурсивный крафт по дереву зависимостей** — промежуточные предметы крафтятся автоматически.
- **Калькулятор потребности** — показывает, чего и сколько не хватает по всему дереву.
- **Спец-механизмы** (печь, доменная печь, коптильня): положить вход → дождаться → забрать результат. Своя вкладка в UI.
- **UI с вкладками** на мониторе, тач-управление: Крафт, Хранилище, Механизмы (динамически), Рецепты, Лог.
- **Локализация**: ID → русские имена через `lang/ru.lua`, красивый fallback, лог отсутствующих переводов.
- **Расчёт времени крафта**: скользящее среднее `avgTime` на рецепт, оценка общего времени по дереву (параллельные/последовательные шаги), ETA и прогресс-бар во время выполнения.
- Работает и в **CraftOS-PC** для отладки.

## Установка

На компьютере или черепахе выполните:

```lua
wget run https://raw.githubusercontent.com/hirokyserega-web/shellcraft/main/install.lua
```

Скрипт спросит роль (`core` или `worker`), скачает все файлы, создаст `config.dat` и настроит автозапуск. После установки перезагрузите компьютер/черепаху (`reboot`) — ShellCraft запустится автоматически.

> Если запускаете с форка — поменяйте `USER` в начале `install.lua`.

## Схема сборки в игре

### Core (главный компьютер)

Поставьте:

- **Computer / Advanced Computer**;
- **wired modem** (для rednet с воркерами и доступа к инвентарям по сети);
- **монитор** (для UI);
- **сундуки/бочки** рядом, подключённые к проводной сети (хранилище);
- при желании **печь / доменную печь / коптильню** — они попадут во вкладку «Механизмы».

Core сам найдёт хранилища, мониторы, модемы, машины и воркеров в сети.

### Worker (черепаха-крафтер)

Каждая воркер-черепаха должна иметь:

- **wired modem** — Worker-черепашка ОБЯЗАТЕЛЬНО подключается к проводному модему (wired modem), и модем должен быть активирован (ПКМ по модему, он должен светиться красным). Беспроводной модем нужен только для обнаружения; перемещение предметов идёт по проводной сети. Core кладёт ингредиенты прямо в инвентарь черепахи и забирает результат через `pushItems`/`pullItems` по проводной сети;
- принадлежность к той же проводной сети, что и Core;
- свободный крафт-инвентарь (слоты 1–16).

Черепахе **не нужны** свои сундуки-буферы — Core манипулирует её инвентарем напрямую по проводному модему.

## Как добавить воркеров

1. Поставьте черепаху, подключите её проводным модемом к сети Core.
2. Выполните на ней команду установки (см. выше), выберите роль `worker` и укажите ID компьютера Core.
3. Перезагрузите черепаху.
4. Она сама зарегистрируется в Core через rednet. Core начнёт раздавать ей задачи параллельно с другими воркерами.

ID Core-компьютера виден на его мониторе в строке статуса или в `config.dat`.

## Обучение рецептов

1. Перейдите во вкладку **Рецепты** на мониторе Core.
2. Нажмите **+ Обучить**.
3. На черепахе-обучателе разложите предметы в слотах 1–9 как в верстаке.
4. Нажмите **Готово** на мониторе.
5. Система выполнит крафт, считает ID и количество результата и сохранит рецепт (`shaped`).

Для рецептов через механизм (печь и т.п.) используйте тип `machine` — Core обработает его сам через вкладку «Механизмы».

## Как работает автообновление

При запуске `startup.lua` сначала вызывает `updater.lua`:

1. читает локальный `version`;
2. сравнивает с `version` на GitHub;
3. если новее — скачивает все `.lua` файлы;
4. **не трогает** защищённые файлы (`config.dat`, `recipes.dat`, `config.local.lua`, `lang/missing.log`, `lang/cache`, `shellcraft.log`);
5. перезагружает компьютер (`os.reboot`).

## Заказ крафта и дерево зависимостей

1. Вкладка **Крафт** → тап по рецепту → выбор количества.
2. Калькулятор показывает потребность по всему дереву: что есть (зелёным) и чего не хватает (красным).
3. **ОК** — Core строит дерево зависимостей и ставит шаги в очередь (от листьев к корню).
4. Промежуточные предметы крафтятся автоматически, финальный — последним.
5. Если базового ресурса не хватает — заказ отклоняется с точным указанием, чего и сколько не хватает.

## Локализация

ID предметов приходят на английском (`minecraft:oak_planks`). Внутри системы (рецепты, хранилище, конфиг, сообщения протокола) **всегда используется только ID как ключ** — имя никогда не служит ключом. Для вывода пользователю есть модуль `lib/names.lua` с функцией `names.display(id, detail)`, которая возвращает имя по приоритету:

1. русское имя из автособранного словаря `lang/ru.lua`;
2. `displayName` из `getItemDetail` (кешируется в `lang/cache`, чтобы не дёргать периферию каждый кадр);
3. fallback: красиво отформатированный ID (`Oak planks`).

Отсутствующие имена (нет ни русского перевода, ни `displayName`) логируются без дублей в `lang/missing.log` — удобно дозаполнить словарь.

### Автогенерация словаря

Словарь `lang/ru.lua` собирается **автоматически** из официальных файлов перевода Minecraft (`ru_ru.json`), а не пишется вручную. Генератор `tools/gen_lang.lua` сопоставляет translation key с item ID:

- `block.<namespace>.<path>` → `<namespace>:<path>`
- `item.<namespace>.<path>` → `<namespace>:<path>`
- приоритет у `block.*` (каноническое имя блока); ключи с лишними сегментами (`item.minecraft.potion.effect.leaping`) пропускаются.

**Где взять `ru_ru.json`:**

- ванилла: из `.jar` Minecraft (например `assets/minecraft/lang/ru_ru.json`) или зеркал вроде [Inventivetalent/lang-mappings](https://github.com/Inventivetalent/lang-mappings);
- моды: внутри `.jar` каждого мода в `assets/<modid>/lang/ru_ru.json` (если мод не поставляет русский — генератор просто не добавит его предметы, они покажутся по fallback).

**Запуск генератора** (работает и в CraftOS, и в обычной Lua 5.1+):

```sh
# ванилла + моды -> один словарь (последний аргумент — выходной файл):
lua tools/gen_lang.lua vanilla/ru_ru.json mod_create_ru_ru.json lang/ru.lua
```

Можно передать несколько входных JSON — они объединятся. Готовый `lang/ru.lua` коммитится в репозиторий, так что пользователь без генератора всё равно получает переводы.

## Расчёт времени крафта

ShellCraft измеряет фактическое время каждой операции крафта и оценивает общее время мегапредмета по дереву зависимостей.

**Замер и скользящее среднее.** При каждом крафте (верстак или механизм) замеряется реальное время (раскладка + операция) и нормализуется на 1 операцию. Рецепт хранит поле `avgTime` — экспоненциальное скользящее среднее (`alpha = 0.3`) по замерам, которое уточняется со временем и сохраняется в `recipes.dat`. Для механизмов учитывается реальное время обработки печи/коптильни.

**Оценка по дереву (`planner.estimateTime`).** Суммируется `avgTime * crafts` всех операций. Независимые задачи (один уровень дерева, параллельные ветви) делятся на число свободных исполнителей (воркеры + машины) и считаются параллельно; зависимые (критический путь) — последовательно. Это даёт оценку критического пути, а не наивную сумму.

**Оценка по умолчанию.** Для нового рецепта, у которого `avgTime` ещё не измерен, берётся разумное значение по умолчанию (крафт ≈ 1 с, механизм ≈ 10 с на операцию). Оценка помечается как приблизительная `(прибл.)` и уточняется после первых реальных прогонов.

**В UI.** При выборе количества крафта сразу показывается оценка `≈ N мин`. Во время выполнения в подвале рисуется прогресс-бар `X/Y` и оставшееся время `ETA`: пока шагов не завершено — используется начальная оценка, затем ETA пересчитывается по фактической скорости.

## Структура репозитория

```
shellcraft/
├── install.lua          — установка одной командой
├── startup.lua          — апдейт + запуск роли (core/worker)
├── updater.lua          — автообновление с GitHub
├── config.lua           — автогенерируемый конфиг + автоопределение периферии
├── version              — версия проекта
├── core/
│   ├── server.lua       — главный сервер (parallel-корутины)
│   ├── storage.lua      — хранилище, pushItems/pullItems
│   ├── recipes.lua      — рецепты + обучение
│   ├── planner.lua      — дерево зависимостей, калькулятор
│   ├── machines.lua     — спец-механизмы (печь и т.п.)
│   └── dispatcher.lua   — раздача задач воркерам
├── worker/
│   └── worker.lua       — черепаха-крафтер
├── ui/
│   ├── ui.lua           — вкладки, отрисовка, тач
│   └── widgets.lua      — рамки, кнопки, списки
├── lang/
│   └── ru.lua           — словарь переводов (автогенерируется)
└── lib/
    ├── net.lua          — rednet-протокол
    ├── util.lua         — логи, файлы, форматирование
    └── names.lua        — отображаемые имена + кеш
└── tools/
    └── gen_lang.lua     — генератор словаря из ru_ru.json
```

## Технические детали

- Только API CC: Tweaked: `turtle`, `peripheral`, `fs`, `os`, `rednet`, `term`, `parallel`, `textutils`, `colors`, `http`.
- Перемещение предметов — `pushItems`/`pullItems` через проводные модемы.
- UI и сетевой слушатель работают параллельно через `parallel.waitForAny`.
- Ошибки (нет ингредиента, воркер недоступен, машина занята) логируются и показываются в UI.

## Fluid & Machine Limitations

- **Turtles cannot carry fluids**: All fluid transfer is performed Core-side via wired modems. Tanks and machines must be connected to the same wired modem network as the Core and enabled (glowing red).
- **Core-Side Routing Only**: ShellCraft only handles item/fluid movement and timing. External power (Create rotation, IC2 EU, heat, RF) must be supplied manually by your build.
- **Energy Display**: Energy is displayed in FE (Forge Energy) only. IC2 EU is generally not readable by ComputerCraft and will show as N/A.
- **Snapshot learning**: The auto-learning snapshot recorder requires the station/machine to finish one full manual processing cycle before you click "Finish & Save".
- **Danks**: A "dank" is a dedicated fluid tank on the wired network reserved for a single fluid. ShellCraft tracks its fill level and uses it as the primary supply source for recipes before falling back to the general pool. All danks configuration is saved in `config.local.lua`.
- **Furnace Fuel**: Automated fueling of vanilla furnaces, blast furnaces, and smokers is out of scope and must be managed manually.
